A ·Concept Cardsसंकल्पना पत्रके

Every Week-1 concept, explained as a one-page artifact: what it is, why it's needed, how it works, its trade-offs, and how our hospital chatbot actually uses it.

1 · Text Preprocessing
Preprocessing

Sentence Segmentation

Definition
Splitting a block of text into individual sentences.
Purpose
Lets the system handle each patient statement as one unit of intent.
Working principle
Detects boundary markers (।, ., ?, !) while avoiding false splits on abbreviations.
Advantages
Cleaner downstream parsing; enables multi-intent detection.
Limitations
Marathi punctuation and code-mixed text (Marathi+English) confuse rule-based splitters.
"मला अपॉइंटमेंट हवी आहे. डॉक्टर कधी उपलब्ध आहेत?" → 2 sentences → 2 separate intents queued.
Preprocessing

Tokenization

Definition
Breaking a sentence into words/sub-words (tokens).
Purpose
Converts free text into discrete units a model can count and score.
Working principle
Whitespace/regex splitting for English; for Marathi, script-aware or sub-word (SentencePiece) tokenizers handle agglutination.
Advantages
Foundational step — every later technique depends on it.
Limitations
Marathi's conjunct characters and suffix-heavy morphology break naive whitespace tokenizers.
"अपॉइंटमेंट बुक करायची आहे" → [अपॉइंटमेंट, बुक, करायची, आहे]
Preprocessing

Case Normalization

Definition
Converting text to a single case (usually lowercase).
Purpose
Prevents "OPD" and "opd" being treated as different words.
Working principle
Simple string transformation; Marathi (Devanagari) has no case, so this mainly matters for code-mixed English tokens like "OPD", "ICU".
Advantages
Reduces vocabulary size, improves match rate.
Limitations
Can lose meaning for acronyms (e.g. "ICU" vs "icu" is fine, but proper nouns can suffer).
"OPD Timing?" → "opd timing?"
Preprocessing

Stop-word Removal

Definition
Removing high-frequency, low-information words.
Purpose
Keeps only the words that carry the patient's actual intent.
Working principle
Filters tokens against a stop-word list (आहे, ला, आणि, is, the, a…).
Advantages
Shrinks feature space; speeds up intent classification.
Limitations
Marathi stop-word lists are less standardized than English; over-removal can strip negation words and flip meaning.
"मला अपॉइंटमेंट हवी आहे" → [अपॉइंटमेंट, हवी]
Preprocessing

Stemming

Definition
Chopping word suffixes to reach a crude root form.
Purpose
Groups variants of a word ("book", "booking", "booked") under one feature.
Working principle
Rule-based suffix stripping (e.g. Porter stemmer for English).
Advantages
Fast, no dictionary needed.
Limitations
Produces non-words; Marathi's rich inflection makes rule-based stemmers unreliable without a dedicated Marathi stemmer.
"booking" / "booked" → "book"
Preprocessing

Lemmatization

Definition
Reducing a word to its dictionary base form using grammar/vocabulary.
Purpose
Same goal as stemming but keeps the result a valid word.
Working principle
Uses POS tags and a morphological dictionary (e.g. Marathi WordNet).
Advantages
More accurate, linguistically valid output.
Limitations
Slower; needs a good Marathi lexical resource, which is scarcer than English ones.
"करायची" → "करणे" (base verb "to do")
Preprocessing

Noise Removal

Definition
Stripping non-content elements: HTML tags, emojis, extra symbols.
Purpose
Keeps only meaningful text for the model.
Working principle
Regex/pattern-based filtering.
Advantages
Reduces vocabulary pollution.
Limitations
Aggressive cleaning can remove useful signals like "🤕" indicating pain/urgency in a WhatsApp-style query.
"अपॉइंटमेंट हवी आहे!!! 🙏🙏" → "अपॉइंटमेंट हवी आहे"
Preprocessing

Text Cleaning

Definition
The umbrella process combining the above steps into a single pipeline.
Purpose
Produces a consistent, model-ready version of raw patient input.
Working principle
Sequential application: segment → tokenize → normalize → destop → stem/lemmatize → denoise.
Advantages
Standardizes messy, real-world chat input.
Limitations
Over-cleaning can erase context a chatbot needs (e.g. urgency markers).
See the live pipeline demo on the home page — that is this step, running on your input.
2 · Feature Engineering
Feature Eng.

Bag of Words (BoW)

Definition
Represents text as word counts, ignoring order.
Purpose
Turns cleaned tokens into a numeric vector for classification.
Working principle
Builds a vocabulary; each document becomes a count vector over that vocabulary.
Advantages
Simple, fast, interpretable.
Limitations
Loses word order — "doctor available" and "available doctor" look identical.
Used as a baseline for our department-routing intent classifier.
Feature Eng.

N-Grams

Definition
Contiguous sequences of N words/characters.
Purpose
Captures local word order BoW loses.
Working principle
Slide a window of size N across tokens (bigrams, trigrams…).
Advantages
Captures phrases like "हृदयरोग तज्ञ" (cardiologist) as one unit.
Limitations
Vocabulary explodes quickly with larger N.
Bigram "हृदयरोग तज्ञ" routes directly to Cardiology, vs. the words scored separately.
Feature Eng.

TF (Term Frequency)

Definition
How often a term appears in a document, normalized by document length.
Purpose
Weights a word by its local importance.
Working principle
TF = (count of term) / (total terms in document).
Advantages
Simple and captures within-query emphasis.
Limitations
Common words like "आहे" still score high unless removed first.
"अपॉइंटमेंट" repeated in a long query scores higher, signalling booking intent.
Feature Eng.

IDF (Inverse Document Frequency)

Definition
Down-weights terms that appear across most documents.
Purpose
Distinguishes generic words from distinctive, topic-specific ones.
Working principle
IDF = log(total docs / docs containing the term).
Advantages
Highlights rare-but-informative words like a department name.
Limitations
Needs a representative corpus of past patient queries to compute well.
"बिलिंग" (billing) is rare across all logged queries → high IDF → strong signal for the billing intent.
Feature Eng.

TF-IDF

Definition
TF × IDF — combined weight balancing local frequency and global rarity.
Purpose
The default numeric representation for classical text classifiers.
Working principle
Multiplies the two prior scores per term, per document.
Advantages
Strong baseline; no training data needed beyond the corpus itself.
Limitations
Still ignores semantics — "doctor" and "physician" remain unrelated.
Our department-routing baseline: TF-IDF + logistic regression on 500 sample queries.
3 · Language Representation
Representation

One-Hot Encoding

Definition
Each word as a sparse vector with a single 1 at its vocabulary index.
Purpose
The simplest way to make words machine-readable.
Working principle
Vector length = vocabulary size; one position is "hot".
Advantages
Trivial to implement, no training required.
Limitations
No notion of similarity — "अपॉइंटमेंट" and "भेट" (both mean "appointment") are unrelated vectors.
A 5,000-word hospital-query vocabulary → 5,000-dimension sparse vectors.
Representation

Word Embeddings

Definition
Dense, low-dimensional vectors that place similar-meaning words close together.
Purpose
Lets the chatbot generalize across synonyms and phrasing.
Working principle
Learned from co-occurrence patterns in large text corpora.
Advantages
Captures semantic similarity; smaller vectors than one-hot.
Limitations
Needs a large corpus; good pretrained Marathi embeddings are scarcer than English.
Embeddings place "अपॉइंटमेंट" near "भेट" and "वेळ बुक करणे" — so all route to the booking module.
Representation

Word2Vec

Definition
A neural method (CBOW/Skip-gram) that learns word embeddings from context.
Purpose
Produces reusable, general-purpose embeddings.
Working principle
Predicts a word from its neighbours (CBOW) or neighbours from a word (Skip-gram).
Advantages
Efficient to train; strong semantic structure (analogies).
Limitations
One fixed vector per word — can't tell "बँक" (bank/riverbank) apart by context.
Pretrained on Marathi Wikipedia/news to seed our intent classifier's embedding layer.
Representation

FastText

Definition
An extension of Word2Vec that represents words as bags of character n-grams.
Purpose
Handles morphologically rich languages and unseen words.
Working principle
A word's vector is the sum of its sub-word n-gram vectors.
Advantages
Critical for Marathi — handles inflected forms like "करायची"/"करायचं" as related.
Limitations
Larger model size; sub-word noise on very short words.
Recognizes unseen inflections of "बुक" (booking-related forms) via shared sub-word units.
Representation

Contextual Embeddings

Definition
Embeddings that change per-sentence based on surrounding context (e.g. BERT-style).
Purpose
Resolves ambiguity that static embeddings can't.
Working principle
A transformer encoder produces a different vector for a word depending on its sentence.
Advantages
State-of-the-art accuracy for intent and entity detection.
Limitations
Compute-heavy; needs a Marathi/multilingual model like MuRIL or IndicBERT.
Distinguishes "बिल" as "billing" vs. an unrelated sense, based on full sentence context.
4 · Language Models
Language Models

Statistical Language Models

Definition
Models that predict word sequences using n-gram probabilities.
Purpose
Early approach to next-word prediction and text scoring.
Working principle
P(word | previous n-1 words) estimated by counting from a corpus.
Advantages
Lightweight, interpretable, fast to build.
Limitations
Poor long-range context; data sparsity for rare Marathi phrase combinations.
Autocomplete suggestions for common query starts like "मला ... हवी आहे".
Language Models

Neural Language Models

Definition
RNN/LSTM-based models predicting text using learned dense representations.
Purpose
Better captures longer context than n-gram models.
Working principle
Sequentially processes tokens, maintaining a hidden state of prior context.
Advantages
Handles longer patient sentences with dependent clauses.
Limitations
Slower to train; struggles with very long sequences (vanishing gradient).
An LSTM intent classifier trained on logged Marathi/English chatbot conversations.
Language Models

Transformer-based Representations

Definition
Self-attention-based models (BERT, GPT-style) that weigh all words in a sentence simultaneously.
Purpose
Current state-of-the-art for understanding and generating text.
Working principle
Self-attention layers let every token attend to every other token, in parallel.
Advantages
Best accuracy; multilingual variants (IndicBERT, MuRIL) support Marathi directly.
Limitations
Heavy compute/memory; overkill for a lightweight FAQ-routing chatbot on limited hospital IT budgets.
Candidate backbone for a future v2 of our chatbot that also generates natural Marathi replies, not just routes queries.
